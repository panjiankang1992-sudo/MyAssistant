## User Scenarios & Testing *(mandatory)*

### User Story 1 - 开发者可运行项目 (Priority: P1)

作为开发团队成员，我希望在 macOS 开发机上完成所有必要的开发环境安装和项目初始化，使跨平台应用能在 Android 模拟器和 macOS 桌面上成功编译运行。

**Why this priority**: 这是所有后续工作的基础——没有可运行的项目骨架，任何功能开发都无法开始。

**Independent Test**: 在新机器上按文档执行环境安装步骤后，启动应用能在 Android 模拟器和 macOS 桌面上看到运行界面。

**Acceptance Scenarios**:

1. **Given** 一台已安装必要开发工具（IDE、平台 SDK）的 macOS 开发机, **When** 按照安装指南完成所有工具链和依赖配置, **Then** Android 模拟器上能运行默认应用
2. **Given** 项目已配置 macOS 桌面目标, **When** 启动 macOS 桌面构建, **Then** macOS 原生窗口中出现应用界面
3. **Given** 项目依赖配置文件已正确设置, **When** 执行依赖安装命令, **Then** 所有依赖成功解析且无版本冲突

---

### User Story 2 - 用户管理日常待办事项 (Priority: P2)

作为用户，我希望在应用中查看和管理当天的待办事项，包括查看列表、标记完成、查看详情，以及通过智能输入和手动表单两种方式添加新待办。

**Why this priority**: 待办管理是产品的核心功能，直接交付用户价值。Phase 2 实现此功能符合 MVP 目标。

**Independent Test**: 打开应用 → 进入待办 Tab → 看到当日待办列表 → 添加新待办 → 标记完成 → 验证状态变化。全程无需依赖记账、随手记或 Copilot 模块。

**Acceptance Scenarios**:

1. **Given** 用户首次打开应用进入待办 Tab, **When** 当日有预设示例待办, **Then** 列表展示每条待办的标题、来源标签、类型标签和时间信息
2. **Given** 用户在待办列表中看到一条未完成待办, **When** 用户点击该待办的复选框, **Then** 待办标记为完成态（划线、降透明度、复选框填充）
3. **Given** 用户点击某条待办, **When** 详情弹窗打开, **Then** 显示标题、来源标签、类型标签、时间信息，以及操作按钮
4. **Given** 用户点击右下角"+"按钮, **When** 新增弹窗打开, **Then** 默认显示"智能新增"面板，包含文本输入框和语音按钮
5. **Given** 用户在智能新增输入框输入"明天下午三点跟产品开会讨论新功能排期", **When** 点击分析或回车, **Then** 系统自动识别出标题、类型（工作）、时间（15:00）、日期（明天），并展示在预览区供用户确认
6. **Given** 用户切换到"手动新增"Tab, **When** 填写标题、类型、日期、时间、来源并保存, **Then** 新待办出现在列表顶部
7. **Given** 用户添加了一条待办并关闭应用, **When** 重新打开应用, **Then** 之前添加的待办仍然存在于列表中
8. **Given** 用户长按某条待办, **When** 出现删除菜单并点击"删除待办", **Then** 该待办从列表中移除

---

### User Story 3 - 用户管理例行待办 (Priority: P3)

作为用户，我希望管理重复性的例行待办模板，以便快速复用（但本阶段仅实现管理功能，自动生成待后续实现）。

**Why this priority**: 例行待办是待办模块的辅助功能，提升用户效率但不如核心 CRUD 紧急。

**Independent Test**: 打开例行管理 → 添加一条例行待办模板 → 验证列表中显示 → 删除例行待办。

**Acceptance Scenarios**:

1. **Given** 用户在待办 Tab 点击设置齿轮按钮, **When** 例行管理弹窗打开, **Then** 显示已有的例行待办列表
2. **Given** 用户在例行管理弹窗点击"添加例行待办", **When** 填写标题、类型、时间并保存, **Then** 新例行待办出现在列表中
3. **Given** 用户在例行列表中点击某条例行待办的删除按钮, **When** 确认删除, **Then** 该例行待办从列表中移除

---

### User Story 4 - 用户与 AI Copilot 交互 (Priority: P2)

作为用户，我希望在 Copilot Tab 中查看 AI 欢迎界面和建议提示词，并能在输入框中输入问题与 AI 对话。

**Why this priority**: Copilot 是产品的差异化功能。Phase 2 实现基础界面和交互框架，为 Phase 3 的 Agent 深度集成奠定基础。

**Independent Test**: 切换到 Copilot Tab → 看到欢迎界面和提示词卡片 → 输入文字并发送 → 获得 AI 回复（或占位回复）。

**Acceptance Scenarios**:

1. **Given** 用户切换到 Copilot Tab, **When** 界面加载完成, **Then** 显示个性化的欢迎问候语和四个建议提示词卡片（分析消费、总结周报、安排日程、提醒待办）
2. **Given** 用户在输入框中输入问题并点击发送, **When** 消息发送成功, **Then** 用户消息出现在对话区域，系统以本地模拟回复回应（预设文本或占位提示）
3. **Given** 用户在 Copilot 输入框旁选择不同的 AI 模型, **When** 切换模型选择, **Then** 后续对话使用所选模型
4. **Given** 用户点击某个建议提示词卡片, **When** 卡片被点击, **Then** 该提示词自动填入输入框（或直接发送）

---

### Edge Cases

- 当智能新增输入的内容无法被识别时（如仅输入"嗯"），系统应提示"未能识别，请尝试更具体的描述"
- 当用户尝试添加空白标题的待办时，应提示"请输入标题"并阻止保存
- 当待办列表为空时，应显示空状态提示（图标 + 文字引导）
- 当 AI 服务不可用时，Copilot 应显示友好的错误提示而非崩溃（本阶段模拟回复不会触发此情况，但 UI 应预留错误状态）
- 当网络断开时，AI 对话应显示"网络不可用，请稍后重试"

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 开发者 MUST 能够按照文档完成跨平台应用开发环境的安装（含必要的平台 SDK 和项目依赖）
- **FR-002**: 项目 MUST 能在 Android 模拟器和 macOS 桌面上成功编译并运行
- **FR-003**: 系统 MUST 展示当日的待办事项列表，每条显示标题、来源标签（推荐/例行/消息/日历）、类型标签（账单/工作/个人/健康）、时间
- **FR-004**: 用户 MUST 能够通过点击复选框标记待办为完成或取消完成
- **FR-005**: 用户 MUST 能够通过点击待办查看详情弹窗，展示完整的来源、类型、时间、详情信息
- **FR-006**: 系统 MUST 提供"智能新增"功能：用户输入自然语言描述，系统自动解析出标题、类型、时间、日期、来源
- **FR-007**: 系统 MUST 提供"手动新增"表单：用户可填写标题、详情、类型、日期、时间、来源
- **FR-008**: 用户 MUST 能够管理例行待办模板（查看、新增、删除）
- **FR-009**: 用户 MUST 能够通过长按或右键删除单条待办
- **FR-010**: 系统 MUST 提供 Copilot Tab，包含欢迎界面、建议提示词卡片、文本输入框、模型选择器和发送按钮
- **FR-011**: 用户 MUST 能够在 Copilot 中输入文字并发送，获得模拟 AI 回复（本阶段不接入真实 AI 后端，回复为本地预设或占位文本）
- **FR-012**: 系统 MUST 在待办列表为空时显示空状态引导
- **FR-013**: 系统 MUST 支持底部导航栏在待办、记账、随手记、Copilot 四个 Tab 之间切换（记账和随手记显示占位界面）
- **FR-014**: 系统 MUST 将待办数据和例行待办模板持久化到本地存储，应用重启后数据不丢失

### Key Entities *(include if feature involves data)*

- **待办 (Todo)**: 标题、详情描述、来源标签（推荐/例行/消息/日历）、类型标签（账单/工作/个人/健康）、时间、日期、完成状态
- **例行待办模板 (Routine)**: 标题、详情描述、类型、时间
- **Copilot 对话消息 (Chat Message)**: 发送者（用户/AI）、内容文本、时间戳、消息类型（思考/工具调用/结果/错误）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新开发者从零开始到成功运行项目的时间不超过 2 小时（按照文档操作）
- **SC-002**: 用户能在 3 次点击内完成一次待办添加（从打开新增弹窗到保存成功）
- **SC-003**: 智能新增的自然语言识别覆盖至少 4 种类型（工作/账单/健康/个人）和 5 种时间表达（今天/明天/后天/下周/具体时间）
- **SC-004**: Copilot 对话从发送消息到收到模拟回复的延迟不超过 1 秒（本地生成，无网络依赖）
- **SC-005**: 所有页面在设备上的首屏渲染时间不超过 2 秒
- **SC-006**: 用户能在 10 秒内理解并开始使用待办核心功能（无需教程）

## Assumptions

- 开发机为 macOS 系统（覆盖 Android + macOS + 鸿蒙三端开发需求）
- 开发者已具备基本的跨平台应用开发知识
- Copilot 在 Phase 2 仅实现 UI 框架和本地模拟回复，不接入真实 AI 后端（完整 AI 集成延后到后续 Phase）
- 记账和随手记 Tab 在 Phase 2 仅展示占位界面，完整功能待后续实现
- 用户界面的视觉设计参考已有的低保真原型
- 数据在 Phase 2 阶段使用本地持久化存储（应用重启后数据保留），后续 Phase 迁移到服务端同步
