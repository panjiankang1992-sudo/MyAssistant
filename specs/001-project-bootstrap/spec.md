## User Scenarios & Testing *(mandatory)*

### User Story 1 - 开发者环境就绪 (Priority: P1)

作为一名开发者，我希望能够在一台新的 macOS 开发机上按照指南完成全部开发环境的搭建，使项目能够成功编译并在 Android 模拟器和 macOS 桌面上运行，以便开始编码工作。

**Why this priority**: 环境搭建是所有后续开发的前置条件。没有可用的开发环境，任何代码都无法编写、编译和运行。这是项目从设计阶段进入实现阶段的第一步。

**Independent Test**: 在一台新 macOS 机器上执行环境搭建步骤，最终执行 `flutter doctor` 全部通过，`flutter run -d macos` 可启动桌面应用。

**Acceptance Scenarios**:

1. **Given** 一台满足最低配置要求的 macOS 开发机，**When** 开发者按照环境搭建文档安装 Flutter 3.41.7、Android Studio、Xcode，**Then** `flutter doctor` 显示 Android 和 macOS toolchain 均为 ✓ 状态。
2. **Given** 环境搭建完成，**When** 开发者创建 Flutter 项目并执行 `flutter run -d macos`，**Then** macOS 桌面窗口成功启动，显示默认 Flutter 应用。
3. **Given** Android 模拟器已启动，**When** 开发者执行 `flutter run -d android`，**Then** 应用在模拟器中成功启动并显示默认界面。
4. **Given** 项目依赖（Riverpod、go_router、Drift 等）已声明在 pubspec.yaml 中，**When** 执行 `flutter pub get`，**Then** 所有依赖成功解析下载，无版本冲突。

---

### User Story 2 - 我的待办（Todo Management）(Priority: P2)

作为一名用户，我希望能够创建、查看、编辑和管理我的待办事项，包括手动添加和智能识别添加两种方式，以便高效组织我的日常任务。

**Why this priority**: 待办管理是 AI 个人助手的核心功能之一（方案文档中明确的 MVP 交付物）。用户通过待办列表管理日常任务，后续 Copilot 将基于待办数据提供智能建议。此功能可独立于 Copilot 运行并交付价值。

**Independent Test**: 用户打开应用，在待办 Tab 页面创建一条待办事项、勾选完成、查看详情、删除待办，所有操作均可独立完成无需其他功能模块。

**Acceptance Scenarios**:

1. **Given** 用户打开应用并处于待办 Tab，**When** 用户点击"+"按钮选择手动新增，填写标题、类型、时间、日期后保存，**Then** 该待办出现在今日待办列表中，显示正确的标题、类型标签、来源标签和时间。
2. **Given** 用户在智能新增输入框中输入"明天下午三点跟产品开会讨论新功能排期"，**When** 用户点击分析按钮，**Then** 系统自动识别：标题为"跟产品开会讨论新功能排期"，类型为"工作"，时间为"15:00"，日期为明天，并在预览区展示识别结果供用户确认。
3. **Given** 待办列表中存在多条待办，**When** 用户点击某条待办的勾选框，**Then** 该待办标记为已完成（视觉上有划线、半透明效果）。
4. **Given** 用户点击某条待办卡片，**When** 详情弹窗弹出，**Then** 显示该待办的完整信息：标题、来源标签、类型标签、时间、详细描述。
5. **Given** 用户长按某条待办卡片，**When** 上下文菜单出现"删除待办"选项，**Then** 用户点击后该待办从列表中移除。
6. **Given** 用户打开例行待办管理面板，**When** 用户添加一条例行待办（如"周报撰写"，每周五 14:30），**Then** 该系统在对应时间自动生成该待办。

### User Story 3 - Copilot AI 助手 (Priority: P3)

作为一名用户，我希望能够通过 Copilot 对话界面与 AI 助手交互，获取智能建议（如消费分析、周报生成、日程规划、到期提醒），并获得流式实时回复体验。

**Why this priority**: Copilot 是 AI 助手的差异化功能，也是方案文档中 Phase 2 的核心交付物。它依赖待办数据才能提供有价值的建议，因此排在待办功能之后。但它是产品的核心卖点。

**Independent Test**: 用户切换到 Copilot Tab，点击推荐提示卡片或输入自定义问题，获得 AI 流式回复。此功能的聊天界面和交互体验可独立验证，AI 回复内容可先使用模拟数据。

**Acceptance Scenarios**:

1. **Given** 用户切换到 Copilot Tab，**When** 页面加载完成，**Then** 显示 AI 助手图标、个性化问候语（含用户名称）、4 个推荐提示卡片（消费分析、周报生成、日程规划、到期提醒）。
2. **Given** 用户点击推荐提示卡片"分析本月消费趋势，给出省钱建议"，**When** AI 开始处理请求，**Then** 聊天界面显示用户消息气泡，随后 AI 以流式方式逐步输出回复内容。
3. **Given** 用户在底部输入框输入自定义问题并发送，**When** AI 收到请求，**Then** 消息出现在对话历史中，AI 开始生成回复（支持思考过程展示、工具调用提示和最终结果展示）。
4. **Given** 用户可在输入框旁的模型选择器中切换模型（如 DeepSeek、Claude 等），**When** 用户选择不同模型后发送消息，**Then** 请求使用所选模型进行处理。
5. **Given** AI 回复过程中用户无需等待全部完成，**When** 回复以 SSE 流式传输，**Then** 用户可实时看到文字逐字出现。

---

### Edge Cases

- 环境搭建时 Flutter 国内镜像不可用怎么办？（应提供备用镜像或离线安装方案）
- 用户智能新增输入无法识别的内容（如纯表情、无意义字符串）怎么办？（显示"无法识别，请手动输入"提示，引导用户切换到手动模式）
- 待办列表为空时显示什么？（显示空状态插图 + 引导文字"暂无待办事项，点击右下角 + 添加"）
- Copilot 服务不可用时用户体验如何？（显示友好的错误提示"AI 助手暂时不可用，请稍后再试"，不阻塞其他 Tab 功能）
- 用户快速连续发送多条消息时如何处理？（排队处理，显示发送状态）
- 待办日期为历史日期时如何展示？（不在今日列表中显示，可在历史记录中查看）
- 用户未登录时 Copilot 是否可用？（不可用，提示用户先登录。但待办的本地操作可用）

## Requirements *(mandatory)*

### Functional Requirements

**环境搭建**

- **FR-001**: 系统必须提供一份完整的环境搭建文档，涵盖 Flutter 3.41.7 安装、Android Studio 配置、Xcode 配置、FVM 版本管理、国内镜像配置。
- **FR-002**: 项目必须初始化 Flutter 工程，包含 android 和 macos 平台支持，配置好 pubspec.yaml 中所有核心依赖（Riverpod、go_router、Drift、Dio、腾讯云 IM SDK 等）。
- **FR-003**: 项目必须配置代码生成工具链（build_runner、freezed、json_serializable、drift_dev、riverpod_generator）。
- **FR-004**: Android 端必须配置 NotificationListenerService 权限声明和前台服务声明。

**我的待办**

- **FR-005**: 用户必须能够手动创建待办事项，包括标题、详情、类型（工作/个人/账单/健康）、日期、时间、来源（推荐/例行/消息/日历）。
- **FR-006**: 用户必须能够通过智能新增功能输入自然语言文本，系统自动解析出标题、类型、时间、日期等信息并展示预览。
- **FR-007**: 智能新增必须能识别中文自然语言中的时间表述（如"明天"、"下午三点"、"下周五"），并正确转换为日期和时间。
- **FR-008**: 智能新增必须能根据关键词推断待办类型（如"会议"→工作、"缴费"→账单、"健身"→健康）。
- **FR-009**: 用户必须能够勾选待办标记为已完成，已完成待办视觉上区分（划线、半透明）。
- **FR-010**: 用户必须能够点击待办查看完整详情（标题、来源、类型、时间、描述）。
- **FR-011**: 用户必须能够删除待办事项（通过长按上下文菜单或详情弹窗）。
- **FR-012**: 系统必须支持例行待办管理——用户可创建周期性模板（标题、类型、时间），系统自动生成对应的待办事项。
- **FR-013**: 待办列表必须按日期筛选，默认显示今日待办。

**Copilot**

- **FR-014**: Copilot 页面必须显示 AI 助手问候语（含用户名称）和推荐提示卡片。
- **FR-015**: 用户必须能够点击推荐提示卡片或输入自定义消息发起 AI 对话。
- **FR-016**: AI 回复必须支持流式输出，用户可以实时看到文字逐字出现。
- **FR-017**: 对话界面必须区分用户消息和 AI 消息（不同对齐方式和样式），支持显示 AI 思考过程和工具调用状态。
- **FR-018**: 用户必须能够切换 AI 模型（至少提供 2 个模型选项）。
- **FR-019**: 消息列表必须支持滚动，新消息自动滚动到底部。
- **FR-020**: 用户在等待 AI 回复时必须看到加载指示器。

### Key Entities

- **待办事项 (Todo)**：标题、描述、类型（工作/个人/账单/健康）、优先级、截止日期、时间、来源（推荐/例行/消息/日历）、完成状态、创建时间、更新时间。
- **例行待办 (Routine)**：标题、描述、类型、触发时间、重复规则。
- **对话消息 (ChatMessage)**：角色（用户/AI）、内容、消息类型（思考/工具调用/结果/错误）、时间戳、会话 ID。
- **AI 推荐提示 (Prompt Suggestion)**：标题、图标、提示文本。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 开发者在 4 小时内可完成从零到 Flutter 项目成功运行（`flutter run -d macos`）的完整环境搭建。
- **SC-002**: 所有 pubspec.yaml 声明的依赖 `flutter pub get` 一次成功，无版本冲突警告。
- **SC-003**: 用户手动创建一条待办的操作流程在 30 秒内完成（从点击"+"到保存成功）。
- **SC-004**: 智能新增对常见中文自然语言输入的识别准确率达到 85% 以上（类型、时间、日期均正确）。
- **SC-005**: 待办列表渲染流畅——100 条待办滚动无明显卡顿。
- **SC-006**: AI 对话流式回复的首字延迟不超过 2 秒。
- **SC-007**: Copilot 页面从切换到可交互状态不超过 1 秒。
- **SC-008**: 用户在无网络时仍然可以进行待办的增删改查操作（离线可用）。

## Assumptions

- 开发者使用 macOS 作为开发机（可同时覆盖 Android 和 macOS 编译）。
- 鸿蒙（HarmonyOS）环境搭建在本次迭代中暂不包含——鸿蒙需要独立的 DevEco Studio 和 Flutter-OH SDK，将在后续迭代中单独处理。
- AI 对话的后端服务已单独开发和部署（本 spec 聚焦客户端实现），Copilot 的 AI 回复在开发阶段可使用模拟数据。
- 待办数据在 V1 阶段仅存储在本地（SQLite），不涉及多端同步和云端存储。
- 用户界面遵循 Apple 设计风格（参考 `todo-app.html` 低保真原型中的设计语言）。
- 单用户场景——V1 不考虑多用户协作。
- 腾讯云 IM SDK 的集成在环境搭建阶段仅做依赖声明和基础初始化，实际消息收发在后续迭代中实现。
- 项目代码组织遵循 `Flutter从零开始技术指导方案.md` 中定义的目录结构（lib/core、lib/domain、lib/data、lib/features）。
